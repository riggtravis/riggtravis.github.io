A New Direction: Maximizing Productivity and Chances at Completion

Travis Rigg's Blogging and Development Extravaganza
2016-11-04
CSS, CSS Frameworks, Cascading Style Sheets, Content Delivery Networks, Documentation, Productivity, Reality

Next week I’m changing the CSS framework that my project uses. This seems crazy
but it’s actually my best chance at successfully finishing my project.

Why do we use frameworks?

The point of a framework is that it handles a large body of the work you would
need to do on your own if you weren’t using a framework. When I was first
learning about frameworks, I had trouble understanding what the difference
between a framework and a library. The way that I think of it now is that a
library is code that gives your code the ability to do something. A framework
is a body of code waiting for you to use your code to make it do something.

Frameworks are great for getting actual things done without having to worry
about doing tasks that someone else has already done for you. Yes, libraries
are great for this too, but the bulk of the work in the creation of your
project is still put into understanding lower level problems than what you are
ultimately trying to tackle. When you have a large project, tackling these
small low level problems will take up more of your time than is ultimately
worth it.

In this context, “large project” should be interpreted as anything that takes
longer than a weekend to create, at least in my mind. Your time is better spent
working on other things.

Are there any downsides to using a framework?

Definitely. You are depending on someone else’s code. 99% of the time, this is
a good thing. The smart way to program is to rely on other people and trust
them. If every piece of software was only written by a single person, our most
advanced software projects might still be pong.

That other 1% of the time can be very messy. A framework falls apart as a
useful part of your toolkit when it doesn’t do what you expect it to. This
might be your own failure, selecting a framework that was ill suited for your
needs because you didn’t understand your own needs, or what that framework is
for. The other possibility is that the framework just doesn’t do what its
developer claimed it would do.

I avoid writing cascading style sheets(CSS) as much as I humanly can. CSS isn’t
something that interests me, and it can take hours just to get something small
and simple to look good. When I was doing the bulk of the writing code for my
senior project, I was using a framework for CSS that I have used in the past
and very happy with. I chose to use this framework because I was already
familiar with it and I wouldn’t need to learn anything new to use it.

It doesn’t deliver the results that I had come to expect though. It no longer
matches its documentation. The way I saw it, there were two options available
to me. I could dig into the code base of the new implementation of the
framework in order to keep using it, or I could start using a new framework. I
chose to go for the latter option.

I am moving my project to use Blazecss rather than what I was using before.
This will be an undertaking, but it will be a comparatively small undertaking.
